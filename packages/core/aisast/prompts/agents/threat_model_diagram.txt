<diagram_requirements>
The diagram must include:

1. ARCHITECTURE MAP — ASCII box-and-arrow diagram showing:
   - All major components (boxes)
   - Data flow arrows between components (use → ← ↑ ↓)
   - Trust boundary markers (use dashed lines ─ ─ ─ or labels like [TRUST BOUNDARY])
   - Threats mapped onto the component they affect, with a category label (from the table matching this
     scan's methodology, below) and severity icon
   - Format for threat labels: THREAT-XXX [X] 🔴 Short Title

2. COMPONENT THREAT MATRIX — table mapping each component to its threats, counts by severity

3. THREAT COVERAGE TABLE — count of threats per category (STRIDE category, LINDDUN category, or
   root attack goal for attack-trees — whichever this scan's methodology used)

4. TOP ATTACK PATHS — prose description of the 3-5 most dangerous threat chains. For attack-trees scans,
   these map directly onto the trees themselves: walk each leaf's "parent_step" chain back to its root
   goal ("category") and render it as a path, noting whether each step was AND- or OR-gated ("gate").

Severity icons:
  🔴 critical   🟠 high   🟡 medium   🟢 low

STRIDE abbreviations used in labels (methodology = "stride"):
  [S] Spoofing
  [T] Tampering
  [R] Repudiation
  [I] Information Disclosure
  [D] Denial of Service
  [E] Elevation of Privilege
  [ASI] AI/Agent Security (OWASP ASI)

LINDDUN abbreviations used in labels (methodology = "linddun"):
  [Lk] Linkability
  [Id] Identifiability
  [Nr] Non-repudiation
  [Dt] Detectability
  [Di] Disclosure of Information
  [Un] Unawareness
  [Nc] Non-compliance
  [ASI] AI/Agent Security (OWASP ASI)

Attack-trees labeling (methodology = "attack-trees"):
  No fixed abbreviation table — use the threat's "tree_id" as the label (e.g. [T1]), and use "category"
  (the root goal name) as the coverage-table row instead of a fixed category name. Group leaves by
  "tree_id" when building the coverage table and architecture map.
</diagram_requirements>
