🎯 What is AISAST?

AISAST is an open source, AI-native Static Application Security Testing framework. Unlike traditional SAST tools that match vulnerability signatures against raw source code, AISAST deploys a sequence of five autonomous agents that first understand the application — its architecture, trust boundaries, data flows, and threat surface — before reporting a single vulnerability.

The result is a small set of verified, evidence-backed findings rather than hundreds of alerts that developers learn to ignore. Every reported vulnerability includes the exact file, line number, code snippet, and traced attack path that prove it is real and reachable.

Why Another SAST Tool?

Most SAST tools have the same problem: they generate hundreds of findings, but only a fraction actually matter. Security teams spend days triaging alerts. Developers get overwhelmed by false positives. Eventually the scanner becomes something everyone ignores.

The issue is not that traditional SAST tools are not finding enough vulnerabilities. The issue is that they lack context. A typical SAST engine may flag string concatenation near a SQL query but has no understanding of where that data came from, whether it is user-controlled, or whether the vulnerable path can actually be reached.

Phase 1 — Architecture Assessment

The first agent explores the codebase the way a security engineer would on their first day. It reads the directory structure, maps HTTP endpoints, WebSocket handlers, CLI entry points, and file inputs. It traces authentication flows, identifies what data comes from untrusted external sources versus trusted internal ones, and documents third-party dependencies. This gets written to SECURITY.md and becomes the context every subsequent phase builds on.

Phase 2 — Threat Modeling (STRIDE + OWASP ASI)

The threat modeling agent reads SECURITY.md and applies the STRIDE framework to the specific architecture it just learned about — not a generic ruleset, but threats derived from how this particular application is built.

If the codebase includes LLM API calls, agent frameworks (LangChain, CrewAI, AutoGen, claude-agent-sdk), tool execution patterns, or memory/session management for AI systems, the agent automatically adds OWASP ASI01 through ASI10 threat categories.

Phase 3 — Threat Model Diagram

An optional phase that generates a visual ASCII architecture diagram showing components, trust boundaries, and threat vectors. Saves to .aisast/THREAT_MODEL_DIAGRAM.md.

Phase 4 — Code Review (Evidence-Based)

The code review agent takes the threat model and searches for proof. For each threat it reads the relevant source files, follows the data flow from input to output, and looks for places where untrusted data reaches a sensitive operation without validation.

Evidence Requirement
The agent is explicitly instructed not to report anything it cannot prove with a specific file path, line number, and actual code snippet. This is what keeps false positives low.

Phase 5 — Report Generation

A final agent transforms the vulnerability list into two output formats: a JSON file (scan_results.json) for programmatic processing and a Markdown report with executive summary, severity breakdown, primary exploit chain, and detailed entries including CWE ID, description, code snippet, and remediation guidance.

🎲 LWRA — Lightweight Risk Adjustment Agent

Finding vulnerabilities is only half the problem. With 30 findings across a production codebase, security teams still face the question: what do we fix first?

The LWRA agent answers that question. It runs after the scan in two stages.

Stage 1 — Environment Intake

Ten questions covering: deployment environment (production / staging / dev), architecture (monolith / microservices / serverless), network exposure, WAF, VPN, firewall, rate limiting, data sensitivity (PII, financial, credentials, health), and cloud provider.

Stage 2 — Code-Driven Exploitability Analysis

Without asking the user anything further, LWRA reads every source file relevant to the vulnerability list. For each finding it determines:

  • Auth gate — is there an authentication check before the vulnerable code?
  • Attack complexity — single direct call or multi-step chain?
  • Source-to-sink reachability — does attacker-controlled input reach the sink?
  • Code path activity — is the vulnerable function reachable from a live route?
  • Known exploit patterns — does it match a CVE class or public PoC?

🔀 PR Review Mode

A full repository scan re-analyzes code that hasn't changed and takes minutes to run — the wrong tool for reviewing a single pull request. aisast pr-review instead scans only the diff, using the baseline SECURITY.md and THREAT_MODEL.json from a prior full scan as context for what the change actually touches.

A dedicated pr-code-review agent reads the diff against that baseline and classifies every change into one of four categories:

ClassificationMeaning
new_threatThe change opens a new attack surface that didn't exist before
threat_enablerMakes an already-known threat exploitable — e.g. removes a guard on a previously theoretical path
mitigation_removalDeletes or weakens an existing security control
regressionReintroduces a vulnerability that was already found and fixed

Usage

bash
# One-time baseline (creates SECURITY.md + THREAT_MODEL.json)
aisast scan .

# Review the diff between two branches
aisast pr-review . --base main --head feature/new-login

# Review a specific commit range, or the last N commits
aisast pr-review . --range abc1234~1..abc1234
aisast pr-review . --last 3

# Merge confirmed PR findings back into the baseline
aisast pr-review . --base main --head HEAD --update-artifacts
Why this matters in CI
PR review is what makes AISAST practical to run on every push instead of just nightly — it's scoped to the diff, so it's fast and cheap enough to gate merges without slowing the team down.

🩹 Auto-Fix — aisast apply

LWRA doesn't just prioritize findings — for many vulnerability classes it also generates an exact code_patch: a before/after line replacement at the vulnerable line. aisast apply takes those patches straight from LWRA_REPORT.json and writes them to disk.

👀

Dry-Run First

--dry-run shows exactly what would change, file by file, without touching anything.

✅

Deterministic Only

Only findings with a concrete, deterministic code fix are eligible for auto-apply — nothing is guessed.

bash
# Preview every available patch without writing anything
aisast apply --dry-run

# Apply them all
aisast apply --all

💾 Delta Scanning & Caching

Re-running a full scan after a small code change shouldn't mean paying for and waiting on all five phases again. AISAST hashes the repository and tracks which phases' outputs are still valid.

🔁

Automatic cache hits

If the repo hash hasn't changed since the last scan, unaffected phases are skipped entirely and prior artifacts are reused.

🎯

Delta scans

When only some files changed, AISAST can re-run code review against just the changed files instead of the whole repository.

bash
# Force a full re-run, ignoring the cache
aisast scan . --no-cache

🧵 Parallel Code Review

Code review (Phase 4) is the most expensive phase — it's where every threat hypothesis gets checked against real source. On large repositories with many identified threats, AISAST can split that phase across multiple parallel workers, each independently reviewing a subset of threats.

bash
# 4 parallel code-review workers instead of 1 sequential pass
aisast scan . --workers 4
Sequential-only paths
Delta scans and the optional threat-model-diagram phase always run sequentially — parallel workers only apply to a full scan's code-review phase.

🚀 Quick Start

Installation

bash 
# Clone and install
git clone https://github.com/outseer-public/AISAST
cd AISAST/packages/core
pip install -e .

# Set your API key
export ANTHROPIC_API_KEY="your-key-here"

# Run guided wizard
./aisast

Direct Scan

bash 
# Full scan
aisast scan /path/to/repo

# With diagram + LWRA risk adjustment
aisast scan /path/to/repo --diagram --lwra

# Quiet mode for CI/CD
aisast scan /path/to/repo --quiet

CI/CD Integration

yaml
# .github/workflows/aisast.yml
name: AISAST Security Scan
on: [push, pull_request]
jobs:
  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install aisast
      - run: aisast scan . --quiet
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}

💻 CLI Reference

AISAST ships four commands. Every command supports --quiet and --debug.

aisast scan

Full repository scan — runs the 5-phase pipeline.

FlagDescription
-m, --modelClaude model to use (sonnet / haiku / opus)
-s, --severityMinimum severity to report
-f, --formatmarkdown / json / text / table
-o, --outputWrite report to a specific file
--subagentRun one phase only (assessment / threat-modeling / threat-model-diagram / code-review / report-generator)
--resume-fromResume a scan starting at a specific phase
--diagram / --no-diagramInclude or skip the optional threat model diagram phase
--lwraRun LWRA risk adjustment after the scan
--workersParallel workers for code review, 1–5 (default 1)
--no-cacheIgnore cached artifacts, re-run every phase
--agentic / --no-agenticForce AI/agentic threat classification on or off
--forceSkip confirmation prompts when overwriting artifacts
--skip-checksBypass prerequisite artifact validation

aisast pr-review

Diff-scoped scan for pull requests — see PR Review Mode above.

FlagDescription
--base / --headCompare two branches or commits
--rangeA specific commit range
--diffReview from a saved patch/diff file
--lastReview the last N commits
--update-artifactsMerge confirmed findings back into the baseline VULNERABILITIES.json

aisast lwra

Run risk adjustment on an existing scan's results without re-scanning code.

FlagDescription
-m, --modelClaude model to use
-o, --outputWrite the LWRA report to a specific file

aisast apply

Apply LWRA's generated code patches — see Auto-Fix above.

FlagDescription
--allApply every available code patch
--dry-runPreview changes without writing files

📄 Output Formats

Every scan writes structured artifacts to .aisast/ regardless of format, then renders the report in whichever format you asked for via --format.

FormatUse case
markdown (default)Human-readable report with executive summary, exploit chains, and detailed findings — saved to scan_report.md
jsonMachine-readable, for feeding into dashboards or other tooling
tableCompact terminal summary — good for a quick glance mid-CI-log
textPlain flat list, no styling — greppable output
bash
aisast scan . --format json --output results.json
aisast scan . --format table
